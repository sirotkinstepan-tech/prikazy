# Prikazy MVP Specification

## 0. Repository Status

Отдельный репозиторий **prikazy** (GitHub: `sirotkinstepan-tech/prikazy`). Не связан с проектом «База данных» / `ocr-document-platform`.

Локальные порты по умолчанию смещены (API `8001`, PostgreSQL `5433`, Redis `6380`, MinIO `9002`/`9003`), чтобы оба проекта можно было запускать на одной машине.

## 1. Architecture

The system is a modular backend service for uploading, storing, OCR-processing, and searching PDF/scanned documents.

Primary components:

- FastAPI application exposes upload, document, OCR, search, and health endpoints.
- PostgreSQL is the source of truth for metadata, processing state, OCR text, extracted fields, audit events, and search indexes.
- S3-compatible object storage stores all binary source files. Local development uses MinIO.
- Redis is the Celery broker for asynchronous OCR jobs.
- Celery worker executes OCR processing independently from API request handling.
- OCR provider is an interface with an initial stub provider. A Tesseract provider may be added later without changing API or persistence contracts.
- Alembic manages all schema changes, including partitioned tables and indexes.

High-level flow:

1. Client uploads a PDF or scan to `POST /documents/upload`.
2. API streams the file, computes SHA-256, stores it in MinIO/S3, and creates DB records.
3. API creates a processing job and enqueues a Celery OCR task.
4. Worker downloads the object, runs OCR through the provider interface, persists OCR output and extracted fields, and updates status.
5. Search API queries PostgreSQL full-text search over OCR results with business filters.

Non-goals for MVP:

- User authentication UI.
- Human-in-the-loop validation workflow beyond status support.
- OpenSearch deployment.
- Real production OCR accuracy guarantees. The OCR engine is pluggable and starts with a deterministic stub.

## 2. Stack

- Runtime: Python 3.12.
- API: FastAPI.
- ASGI server: Uvicorn.
- ORM/DB: SQLAlchemy 2.x.
- Migrations: Alembic.
- PostgreSQL driver: psycopg 3.
- Queue/jobs: Celery.
- Broker: Redis.
- Object storage: MinIO through S3 API using `boto3`.
- Config: `pydantic-settings`, environment-based.
- Tests: pytest.
- HTTP test client: FastAPI test client / httpx where appropriate.
- Lint/format: ruff + black.
- Containers: Docker Compose.

Driver decision:

- Use psycopg 3 with SQLAlchemy's synchronous engine for MVP. This keeps the API, repositories, tests, and Alembic setup simpler and aligns well with Celery workers. FastAPI can still serve concurrent requests because DB work is short and can be moved behind service/repository boundaries. If later throughput requires async DB access, the repository interfaces can be migrated to asyncpg without changing API contracts.

## 4. Data Model

All IDs are UUIDs generated by the application unless a table design requires a composite key for partitioning.

### `storage_objects`

Purpose: durable metadata for files stored in S3/MinIO.

Columns:

- `id uuid primary key`
- `bucket text not null`
- `object_key text not null`
- `version_id text null`
- `sha256 text not null`
- `size_bytes bigint not null`
- `mime_type text not null`
- `original_filename text null`
- `created_at timestamptz not null`

Indexes/constraints:

- Unique `(bucket, object_key, coalesce(version_id, ''))` via expression or practical equivalent.
- B-tree on `sha256`.
- Optional deduplication lookup by `(tenant_id, sha256)` is intentionally not placed here because storage objects may be tenant-neutral; document-level dedupe is handled in service logic.

### `documents`

Purpose: business card for a document and processing status.

Partitioning:

- Partitioned by range on `created_at` monthly.
- PostgreSQL requires unique/primary keys on partitioned tables to include the partition key. Use composite primary key `(id, created_at)`.

Columns:

- `id uuid not null`
- `created_at timestamptz not null`
- `tenant_id uuid not null`
- `storage_object_id uuid not null`
- `status text not null`
- `doc_type text null`
- `document_date date null`
- `counterparty_name text null`
- `title text null`
- `source_filename text null`
- `mime_type text not null`
- `size_bytes bigint not null`
- `sha256 text not null`
- `updated_at timestamptz not null`
- `archived_at timestamptz null`

Indexes:

- B-tree `(tenant_id, created_at desc)`.
- B-tree `(tenant_id, status)`.
- B-tree `(tenant_id, doc_type)`.
- B-tree `(tenant_id, document_date)`.
- B-tree `(tenant_id, counterparty_name)`.
- B-tree `(sha256)`.

Status values:

- `uploaded`
- `queued`
- `processing`
- `processed`
- `validated`
- `failed`
- `archived`

### `document_pages`

Purpose: page-level metadata and optional page image references.

Columns:

- `id uuid primary key`
- `document_id uuid not null`
- `document_created_at timestamptz not null`
- `page_number integer not null`
- `storage_object_id uuid null`
- `width integer null`
- `height integer null`
- `created_at timestamptz not null`

Indexes/constraints:

- Unique `(document_id, document_created_at, page_number)`.
- Foreign key to `documents(id, created_at)`.

### `ocr_results`

Purpose: OCR text, layout, confidence, and search vector.

Partitioning:

- Partitioned by range on `processed_at` monthly.
- Use composite primary key `(id, processed_at)`.

Columns:

- `id uuid not null`
- `processed_at timestamptz not null`
- `document_id uuid not null`
- `document_created_at timestamptz not null`
- `job_id uuid not null`
- `provider text not null`
- `language text null`
- `full_text text not null`
- `confidence numeric(5, 4) null`
- `layout_json jsonb null`
- `page_data jsonb null`
- `search_vector tsvector generated always as (to_tsvector('simple', coalesce(full_text, ''))) stored`
- `created_at timestamptz not null`

Indexes:

- GIN on `search_vector`.
- GIN on `layout_json` when present and justified by query paths.
- B-tree `(document_id, document_created_at, processed_at desc)`.
- B-tree `(job_id)`.

Language decision:

- Use PostgreSQL `simple` text search configuration for MVP because documents may contain mixed Russian/English text. This avoids incorrect stemming assumptions. A future migration can add per-tenant or per-document language-specific vectors.

### `extracted_fields`

Purpose: normalized key-value fields extracted from OCR.

Columns:

- `id uuid primary key`
- `document_id uuid not null`
- `document_created_at timestamptz not null`
- `ocr_result_id uuid not null`
- `ocr_result_processed_at timestamptz not null`
- `field_name text not null`
- `field_value text not null`
- `field_type text null`
- `confidence numeric(5, 4) null`
- `source_json jsonb null`
- `created_at timestamptz not null`

Indexes:

- B-tree `(document_id, document_created_at)`.
- B-tree `(field_name, field_value)`.
- GIN on `source_json` only if needed by implemented queries.

### `processing_jobs`

Purpose: application-level job tracking separate from Celery internals.

Columns:

- `id uuid primary key`
- `document_id uuid not null`
- `document_created_at timestamptz not null`
- `job_type text not null`
- `status text not null`
- `attempt integer not null default 0`
- `max_attempts integer not null default 3`
- `celery_task_id text null`
- `error_code text null`
- `error_message text null`
- `locked_at timestamptz null`
- `started_at timestamptz null`
- `finished_at timestamptz null`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Indexes:

- B-tree `(status, created_at)`.
- B-tree `(document_id, document_created_at)`.
- B-tree `(celery_task_id)`.
- B-tree `(job_type, status)`.

### `processing_events`

Purpose: immutable audit trail for document processing.

Columns:

- `id uuid primary key`
- `document_id uuid not null`
- `document_created_at timestamptz not null`
- `job_id uuid null`
- `event_type text not null`
- `message text null`
- `payload jsonb null`
- `created_at timestamptz not null`

Indexes:

- B-tree `(document_id, document_created_at, created_at)`.
- B-tree `(job_id, created_at)`.
- GIN on `payload` only for fields that become queryable.

## 5. API Contracts

All endpoints return JSON except file upload payloads.

### `GET /health`

Returns application health and dependency status where cheap to check.

Response:

```json
{
  "status": "ok",
  "service": "prikazy"
}
```

### `POST /documents/upload`

Multipart form:

- `file`: required PDF/image file.
- `tenant_id`: required UUID.
- `doc_type`: optional string.
- `document_date`: optional date.
- `counterparty_name`: optional string.
- `title`: optional string.
- `idempotency_key`: optional string.

Behavior:

- Validate MIME type and size.
- Compute SHA-256 while storing/streaming.
- Save binary to S3/MinIO.
- Create `storage_objects`, `documents`, `processing_jobs`, and `processing_events`.
- Enqueue OCR task.
- Return immediately; OCR is not blocking.

Response `201`:

```json
{
  "document_id": "uuid",
  "created_at": "timestamp",
  "status": "queued",
  "job_id": "uuid"
}
```

Idempotency:

- If `idempotency_key` is supplied, store and reuse the upload result for the same tenant/request in a future enhancement.
- MVP deduplication checks `(tenant_id, sha256)` at document level and records the existing relation in events, but still allows multiple document records unless product requirements later demand strict dedupe.

### `GET /documents/{id}`

Query:

- `tenant_id`: required UUID.

Response:

- Document metadata, status, storage metadata summary, latest job summary, and latest OCR summary if present.

### `GET /documents`

Query filters:

- `tenant_id`: required UUID.
- `doc_type`: optional.
- `status`: optional.
- `document_date_from`, `document_date_to`: optional.
- `created_at_from`, `created_at_to`: optional.
- `counterparty_name`: optional partial match.
- `limit`: default 50, max 200.
- `offset`: default 0.

### `POST /documents/{id}/reprocess`

Query/body:

- `tenant_id`: required UUID.
- Optional `reason`.

Behavior:

- Create a new `processing_jobs` row.
- Set document status to `queued`.
- Enqueue OCR task.

### `GET /documents/{id}/ocr`

Query:

- `tenant_id`: required UUID.

Response:

- Latest OCR result for the document, including full text, confidence, provider, layout JSON, page data, and extracted fields.

### `GET /search`

Query:

- `tenant_id`: required UUID.
- `q`: required search string.
- `doc_type`: optional.
- `status`: optional.
- `document_date_from`, `document_date_to`: optional.
- `created_at_from`, `created_at_to`: optional.
- `counterparty_name`: optional.
- `limit`: default 20, max 100.
- `offset`: default 0.

Behavior:

- Use `websearch_to_tsquery('simple', q)`.
- Rank with `ts_rank(search_vector, query)`.
- Join latest OCR result to documents.
- Apply tenant and business filters.

Response:

```json
{
  "items": [
    {
      "document_id": "uuid",
      "status": "processed",
      "doc_type": "invoice",
      "document_date": "2026-05-14",
      "counterparty_name": "Example LLC",
      "rank": 0.42,
      "snippet": "..."
    }
  ],
  "limit": 20,
  "offset": 0,
  "total": 1
}
```

## 6. OCR Pipeline

Pipeline states:

1. `uploaded`: file and document metadata created.
2. `queued`: OCR job created and task submitted.
3. `processing`: worker started processing.
4. `processed`: OCR result saved successfully.
5. `validated`: reserved for later manual or automated validation.
6. `failed`: OCR failed after retry policy or unrecoverable error.
7. `archived`: document hidden from active workflows.

Worker steps:

1. Load `processing_jobs` by ID.
2. Acquire job by status transition and lock timestamp.
3. Load document and storage object metadata.
4. Download object from S3/MinIO to temporary worker storage.
5. Run OCR provider.
6. Persist `ocr_results`, `document_pages` if available, and `extracted_fields`.
7. Update `documents.status`.
8. Update `processing_jobs.status`, attempt counters, timestamps, and error fields.
9. Write `processing_events` for each major transition.
10. On retryable failure, record event and let Celery retry.
11. On terminal failure, mark job and document as `failed`.

Provider interface:

- Input: local file path, MIME type, optional language/context.
- Output: full text, confidence, layout JSON, page data, extracted fields.

Initial provider:

- Stub provider that returns deterministic text containing filename/SHA-derived metadata for reliable tests and local end-to-end flow.

Optional future provider:

- Tesseract adapter isolated behind the same interface.

## 7. Queues and Background Jobs

Celery queues:

- `ocr`: default OCR processing queue.

Task:

- `process_ocr_job(job_id: str)`.

Retry policy:

- Max 3 attempts.
- Exponential backoff.
- Retry on storage/network/provider transient errors.
- No retry for validation errors such as unsupported MIME type.

Job tracking:

- Celery is only the delivery mechanism.
- `processing_jobs` is the durable application job state.
- `processing_events` is the immutable audit log.

## 8. Object Storage

Storage rules:

- Do not store PDF/scans in PostgreSQL.
- Store binary files in S3-compatible object storage.
- Local development uses MinIO.
- DB stores bucket, object key, optional version ID, SHA-256, MIME type, size, and original filename.

Object key strategy:

```text
tenants/{tenant_id}/documents/{yyyy}/{mm}/{document_id}/{sha256}-{safe_filename}
```

Reasoning:

- Tenant prefix supports future lifecycle policies and operational debugging.
- Date prefix avoids excessively large flat key listings.
- Document ID avoids accidental object-key collisions.
- SHA-256 supports integrity checks and dedupe workflows.

## 9. Search

MVP search uses PostgreSQL full-text search:

- `ocr_results.full_text` stores raw OCR text.
- `ocr_results.search_vector` is a generated stored `tsvector`.
- GIN index on `search_vector`.
- Queries use `websearch_to_tsquery`.
- Ranking uses `ts_rank`.

Filters:

- `tenant_id`
- `doc_type`
- `status`
- `document_date`
- `created_at`
- `counterparty_name`

Future OpenSearch migration:

- Keep search code behind a repository/service boundary.
- PostgreSQL remains source of truth.
- Future indexer can consume `ocr_results` and `processing_events`.
- API contract should not expose PostgreSQL-specific behavior.

## 10. Migrations and Partitions

Alembic is mandatory for all schema changes.

Partition strategy:

- `documents`: range partition by `created_at`, monthly.
- `ocr_results`: range partition by `processed_at`, monthly.

Implementation decision:

- Use SQLAlchemy models for application mapping.
- Use Alembic `op.execute(...)` raw SQL for partitioned table creation, generated columns, partition DDL, and specialized indexes where SQLAlchemy/Alembic abstractions are insufficient or unclear.
- This is explicit and maintainable for PostgreSQL-specific partitioning behavior.

Partition utility:

- `app/db/partitions.py` exposes a service to create monthly partitions ahead of time.
- It should support both tables.
- Default behavior creates partitions for current month plus N future months.
- Tests cover generated partition names and SQL execution behavior.

Initial partitions:

- Migration creates current month and at least the next two monthly partitions for `documents` and `ocr_results`.
- Runtime utility can be called from startup or scheduled maintenance later.

## 11. Observability

- Request ID middleware can be added if time permits.
- Health endpoint returns basic service health.

Future observability:

## 12. Security

MVP controls:

- Tenant isolation enforced by explicit `tenant_id` filters in all read/search endpoints.
- File MIME type allowlist.
- Upload size limit.
- Safe object key generation; never trust raw filename as a path.
- Do not expose MinIO/S3 internal credentials through API responses.
- Store secrets only in environment variables.
- `.env.example` documents required config without real secrets.
- Basic error responses avoid leaking stack traces.

Deferred controls:

- Authentication and authorization.
- Per-tenant quotas.
- Antivirus/malware scanning.
- PII redaction.
- Signed download URLs.

## 17. Risks and Open Decisions

Risks:

- PostgreSQL partitioned-table foreign keys and composite keys add complexity. The implementation must be tested against real PostgreSQL, not only SQLite.
- Generated `tsvector` columns and partitioned indexes require raw SQL migrations.
- OCR engine quality is not solved by the MVP stub provider. The architecture allows replacing it later.
- Tenant isolation is parameter-based in MVP; production needs authentication and authorization.
- Full-text search in PostgreSQL is sufficient for MVP but may need OpenSearch once ranking, highlighting, language analysis, or scale requirements grow.

Open decisions for user confirmation:

- Should strict deduplication reject duplicate uploads per tenant, or allow multiple document records pointing to identical content?
- Should MVP include real Tesseract integration now, or keep the deterministic stub provider only?
- Which maximum upload size should be enforced by default?
- Which document types should be accepted initially beyond PDF, PNG, JPEG, and TIFF?
